# Wedding Planning App PRD

## Problem Statement

Planning an Indian/Hindu wedding involves many moving parts across Events, Vendors, Rituals, budgets, guests, family responsibilities, documents, payments, and deadlines. The work is currently easy to scatter across chats, spreadsheets, notes, and memory, which makes it hard for the bride and her planning team to know what is done, what is blocked, what is due, and who owns each part.

The app should give the bride and her Collaborators one private planning workspace for her Wedding, with task-driven Progress, clear ownership, in-app reminders, budget/payment tracking, guest planning records, and a Dashboard that highlights urgent action.

## Solution

Build a single-wedding planning app for the bride's Wedding. The app is invitation-only for Collaborators and does not allow Guests to sign in. It organizes planning around one Wedding made up of Events, with Tasks as the central unit of work. Progress rolls up from Task Status across Events, Vendors, Planning Areas, Invitations, Rituals, Items, and the Wedding as a whole.

The first screen is a Dashboard focused on urgent action: upcoming Tasks and Payments, blocked work, overdue work, Wedding and Event Progress, budget snapshot, Vendor pipeline, invite/RSVP snapshot, and the current Collaborator's assignments.

## User Stories

2. As the bride, I want the app to focus on one Wedding, so that I do not have to manage workspaces or choose between weddings.
3. As the bride, I want to invite Collaborators by email, so that trusted family and friends can help plan.
4. As the bride, I want Collaborators to have roles, so that people only have the access they need.
5. As an Owner, I want full control of the Wedding, so that I can manage settings, collaborators, and deletion.
6. As a Planner, I want to manage Events, Tasks, Vendors, Guests, budgets, and documents, so that I can handle major planning work.
7. As a Contributor, I want to update assigned Tasks and statuses, so that I can help without changing the Wedding structure.
8. As the bride, I want Guests to be planning records only, so that invitees do not access the platform.
9. As a Planner, I want to create Events such as Mehndi, Haldi, Sangeet, Ceremony, and Reception, so that the Wedding reflects a multi-event celebration.
10. As a Planner, I want each Event to have its own date, location, Vendors, Tasks, Guests, budget, and Notes, so that each Event can be planned independently.

15. As a Planner, I want to add a blocked reason to a Task, so that others know what is preventing progress.
16. As the bride, I want Wedding Progress to be task-driven, so that completion percentages are explainable.
17. As the bride, I want Progress by Event, so that I can see which Events are falling behind.
18. As the bride, I want Attention Needed signals, so that concrete problems are visible before they become emergencies.

21. As the bride, I want in-app Reminders, so that important work is visible without relying on external notifications.

24. As a Planner, I want to attach Tasks to Vendors, so that booking and follow-up work is tracked.
25. As a Planner, I want to attach Documents to Vendors, so that quotes, contracts, menus, and receipts are easy to find.

28. As the bride, I want a whole-Wedding Budget, so that total planned spending is visible.
29. As the bride, I want Event-level Budgets, so that each Event can be planned and evaluated separately.
30. As a Planner, I want Budget Items, so that expected costs can be organized by category or purpose.
31. As a Planner, I want Payments, so that actual paid and due amounts are tracked separately from planned budget.
32. As the bride, I want payment statuses like Due, Scheduled, Paid, Overdue, and Cancelled, so that cash obligations are clear.
33. As the bride, I want a budget snapshot, so that I can compare budget, expected spend, paid, remaining, and over-budget areas.
34. As a Planner, I want to record Guests, so that invitation and attendance planning can happen inside the app.
35. As a Planner, I want Guest Groups, so that households, families, or shared invitation groups can be managed together.

39. As a Planner, I want manual RSVPs, so that Collaborators can record responses gathered outside the app.
40. As a Planner, I want event-level RSVPs, so that a Guest can attend one Event and decline another.
41. As a Planner, I want Meal Preferences per Guest, so that catering counts and dietary needs are visible.

## Implementation Decisions

- The expected v1 stack is Next.js, React, TypeScript, Supabase Auth, Supabase Postgres, and Supabase Storage.
- The app is scoped to one specific Wedding for v1. There is no multi-wedding workspace model, tenant switching, public signup, or self-serve wedding creation.
- The top-level planning object is the Wedding. Events, Vendors, Guests, Guest Groups, Budgets, Payments, Planning Areas, Rituals, Documents, Notes, Items, Accommodation, Transport, Templates, and Collaborators belong to that Wedding.
- Wedding planning data must be durable in Supabase Postgres. Documents must be stored in Supabase Storage or represented as external links.
- If the project runs on the Supabase Free plan, include a lightweight daily Postgres heartbeat so the project has regular user database activity. A scheduled Edge Function that only returns "hello world" is not sufficient; the heartbeat must read or update Postgres, such as updating a single `app_heartbeats.last_seen_at` row.
- Supabase Pro is the only supported way to guarantee the project is never automatically paused for inactivity. The Free-plan heartbeat is a best-effort cost-saving measure, not a production availability guarantee.
- Guests are not platform users. They are planning records only.
- Collaborator access is invitation-only. The initial bride user is an Owner and can invite other Collaborators.
- Collaborator roles are Owner, Planner, and Contributor.
- Tasks are the central unit of work and the primary basis for Progress calculations.
- Task Status values are Not Started, In Progress, Blocked, and Done.
- Larger planning records can have a Lead, while specific Tasks can have an owner.
- Progress rolls up from Tasks to Events, Vendors, Planning Areas, Invitations, Budget Items, and the Wedding.
- The Dashboard prioritizes urgent action first: upcoming deadlines, overdue work, blocked items, Attention Needed, then overall Progress and financial/invite summaries.
- Vendors use one generic model with required Vendor Category rather than separate models for venue, caterer, decorator, photographer, priest, planner, and transport provider.
- Vendor Status values are Researching, Contacted, Quote Received, Shortlisted, Booked, Rejected, and Cancelled.
- Budgets exist at both whole-Wedding and Event levels.
- Budget Items represent planned or allocated costs. Payments represent actual amounts due or paid.
- Invitations are planning records only. The platform does not send invitations in v1.
- RSVPs are manually recorded by Collaborators and can vary per Event.
- Seating Plans are lightweight planning records, not a drag-and-drop floorplan editor.
- Rituals are first-class but lightweight, because Hindu ceremony planning has ceremony-specific materials, people, timing, and order.
- Documents can be files or external links and can attach to major planning records.
- Notes are freeform planning context attached to major planning records.
- In-app Reminders are in scope. External email, SMS, WhatsApp, and push notifications are out of scope for v1.
- Accommodation and Transport are planning records only. The platform does not book lodging, flights, or rides.
- Planning Areas provide flexible organization for topics that are not naturally Events or Vendors.
- Templates and Checklist Templates are editable starting points, not locked workflows.

## Testing Decisions

- Test from the highest useful seam: authenticated user workflows against the application boundary, using a seeded single Wedding and realistic Collaborator roles.
- Prioritize behavior over implementation details. Tests should assert visible outcomes and persisted planning state rather than internal component structure.
- The core acceptance seam should cover creating and managing the Wedding planning graph: Events, Tasks, Vendors, Budgets, Payments, Guests, RSVP records, Rituals, Documents, Notes, Items, and Dashboard rollups.
- Add focused domain tests for Progress calculations, Attention Needed generation, permission enforcement, RSVP rollups, budget/payment summaries, and Vendor Status transitions.
- Add permission tests proving Owners, Planners, and Contributors can only perform allowed actions.
- Add dashboard tests proving overdue Tasks, blocked Tasks, due Payments, Event Progress, Budget summaries, Vendor pipeline, RSVP snapshots, and current-user assignments are calculated correctly.
- Add template application tests proving templates create editable Events, Tasks, Vendor Categories, Budget Items, Rituals, and Planning Areas.
- Add regression tests for event-level RSVPs, because attendance differs per Event and should not collapse into a single Wedding-level answer.

## Out of Scope

- Multiple Weddings per account or workspace.
- Public signup and self-serve Wedding creation.
- Guest login, guest portals, or guest-facing RSVP pages.
- Sending invitations by email, SMS, WhatsApp, or push notification.
- External notification delivery for reminders.
- Direct booking of hotels, flights, rides, venues, or vendors.
- Payment processing or vendor payouts.
- Drag-and-drop floorplan editing for seating.
- Full activity feed/history.
- General marketplace discovery for Vendors.
- Multi-tenant SaaS concerns such as billing, organizations, workspace switching, and public onboarding.

## Further Notes

- The product should feel like a private planning command center for the bride and trusted Collaborators, not a public wedding website.
- The first build should favor clarity, ownership, and rollups over automation.
- The Dashboard should always answer three questions: what needs attention, how close are we, and who owns the next action?
- Indian/Hindu wedding specificity should come through Templates, Rituals, event structure, vendor categories, guest planning, meal preferences, and logistics support.

## Build Plan

### Phase 1: Foundation

- Create the application shell with invited Collaborator authentication.
- Seed or create the single Wedding.
- Add Supabase persistence for the Wedding record and configure a lightweight daily Postgres heartbeat when running on the Supabase Free plan.
- Implement Collaborator roles: Owner, Planner, Contributor.
- Implement core records: Event, Task, Vendor, Budget, Budget Item, Payment, Guest, Guest Group, RSVP, Document, Note.
- Implement basic Dashboard cards for Tasks due soon, overdue Tasks, blocked Tasks, due Payments, and Event Progress.

### Phase 2: Planning Depth

- Add Vendor Categories, Vendor Status workflow, and Vendor pipeline summary.
- Add Event-level Budgets and budget/payment rollups.
- Add Invitation and RSVP tracking with event-level attendance.
- Add Guest Segments and Meal Preferences.
- Add Documents and Notes across major records.

### Phase 3: Indian/Hindu Wedding Fit

- Add editable Wedding Templates for common Events and starter planning structure.
- Add Checklist Templates for Vendors, Events, Rituals, Planning Areas, and Items.
- Add Rituals with timing, materials, participants, Notes, and related Tasks.
- Add Planning Areas for outfits, decor, food, gifts, logistics, music, beauty, and guest experience.
- Add Items for physical things to buy, pack, deliver, or set up.

### Phase 4: Logistics and Overview

- Add Timeline view across Events, Rituals, Tasks, Payments, and schedule items.
- Add Accommodation planning records.
- Add Transport planning records.
- Add lightweight Seating Plans.
- Add richer Attention Needed signals for missing venue, missing contract, budget overrun, overdue payment, blocked work, RSVP follow-up, and rituals missing Items.

### Phase 5: Polish

- Add search and filtering across planning records.
- Add reminder dates and in-app reminder surfaces.
- Add dashboard drill-downs from every summary card.
- Add import/export paths for guest lists and documents if needed.
- Tighten mobile layouts for quick planning updates during errands and family discussions.
